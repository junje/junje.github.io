Zookeeper介绍

Zookeeper是什么？

Zookeeper是一个分布式应用协调框架，在分布式系统架构中有广泛的应用场景。

Zookeeper能用来干什么？

  1. 配置中心，如到家的鲁班系统
  2. 分布式锁
  3. 注册中心，Dubbo的注册中心
  4. Master选举
  5. ….

数据模型

Zookeeper的数据模型是层次模型，层次模型常见于文件系统。Zookeeper的层次模称作Data Tree是一个树形结构，每个节点叫做ZNode，每个ZNode都可以存储数据，并且有自己的Version。

ZNode节点类型：

  1. 持久化节点
  2. 临时节点
  3. 持久化有序节点
  4. 临时有序节点
  5. 容器节点 3.5.5版本特性
  6. TTL节点 3.5.5版本特性

public class DataNode implements Record {
    DataNode parent; // 父节点
    byte[] data; // 数据
    Long acl; // 权限
    public StatPersisted stat; // 节点状态类 很重要
    private Set<String> children = null; // 子节点
}

public class StatPersisted implements Record {
    // zxid是全局递增的事务ID
    private long czxid; // 节点创建的zxid
    private long mzxid; // 节点最后更新的zxid
    private long ctime; // 节点创建时间
    private long mtime; // 节点更新时间
    private int version; // 节点版本 做CAS乐观锁操作用
    private int cversion;  // 子节点版本
    private int aversion; // 节点的ACL版本
    private long ephemeralOwner; // 创建临时节点的sessionID，临时节点才会用这个属性
    private long pzxid; // 子节点目录最后被更新的zxid，子节点数据更新这个值不变
}

集群角色

  1. Leader。zookeeper集群内只能有一个leader，只有leader负责写操作。
  2. Follower。参与选举流程。
  3. Observer。不参与选举。目的为了提高读取速度。数据只存内存内，不写入磁盘。

Wathcher

Zookeeper可以在节点或目录上注册Watcher，就是监听器，当节点或目录发生变化会发送通知给注册了Watcher的客户端，如目录增加或删除子节点、节点数据变更、Session断链等。

注：低版本的Zookeeper的监听都是一次性的，监听触发一次后就会失效，需要循环注册。3.6.0后支持永久监听。

ACL

ACL负责Zookeeper的权限控制，Zookeeper可以给节点加各种权限，客户端有权限才可以操作节点。

public Vote(long id, long zxid) {
    this.id = id; // myid
    this.zxid = zxid; // 事务ID |--- 32bit 任期 ---|-- 32bit 事务计数器 --|
    this.electionEpoch = -1L; // 选票轮次 每次新投票都+1
    this.peerEpoch = -1L; // 被推举的Leader的epoch
    this.state = ServerState.LOOKING; // 当前服务器的状态
}

选票PK规则

1.比较选举轮次，比自己大的话，就清空自己票箱。 2.比较zxid，zxid大的票胜出。 2.如果zxid相等，比较myid，myid大的胜出。

选举流程

  1. 每个Server启动后会创建一张选票，选票内容是投给自己的。并广播给集群内的其他Follower包括自己。
  2. 按选票规则PK，如果PK失败，改变自己的票。
  3. 统计集群内的票。找出票数大于一半的Server，并设置为leader。如果没有Server票数大于一半，就继续1和2。
  4. 找到leader后其他Server变为follower，不包含Observer。
